## 1. Project Overview

**What is it?** A personal mood/activity analytics REST API built with Go's standard library (`net/http`), no frameworks. Users register, log entries with mood scores (1-10) and categories, and query their data back with pagination.

**Tech stack:**
| Component | Technology | Why |
|-----------|-----------|-----|
| HTTP Server | `net/http` (stdlib) | Learn Go fundamentals, no magic |
| Database | SQLite via `modernc.org/sqlite` | Pure Go, no CGO, single-file DB |
| Cache / Rate Limiting | Redis via `go-redis/v9` | Shared state across restarts |
| Auth | JWT (`golang-jwt/v5`) + bcrypt | Industry-standard stateless auth |
| Config | `.env` via `godotenv` + `os.Getenv` | 12-factor app pattern |
| Logging | `log/slog` (stdlib) | Structured JSON logging, production-ready |

**Architecture style:** Clean-ish layered architecture with clear separation:
```
HTTP Request → Middleware Chain → Handler → DB/Cache → HTTP Response
```

### Why This Structure Matters (Go Conventions)

| Convention | What You Did | Why It's Correct |
|------------|-------------|------------------|
| `cmd/server/main.go` | Entry point under `cmd/` | Standard Go project layout. If you add a CLI tool later: `cmd/cli/main.go` |
| `internal/` | All business logic here | Go **enforces** that `internal/` packages cannot be imported by other modules. This is compile-time encapsulation — not just convention |
| One package = one responsibility | `db/` does DB, `cache/` does caching | Each package has a clear, testable boundary |
| `models/` separated from `db/` | Structs define shape, `db/` handles persistence | Change DB engine → only `db/` changes. Models are stable |
| Flat package structure | No `internal/handlers/v1/` nesting | Correct for this size. Go prefers flat over deep |

### Dependency Flow (Who Imports Who)

```
main.go
  ├── config     (loads env vars)
  ├── logger     (initializes slog)
  ├── db         (opens SQLite)
  ├── redis      (connects to Redis)
  ├── cache      (imported by main for shutdown)
  ├── worker     (starts goroutine pool)
  └── handlers   (registers HTTP routes)
        ├── db        (CRUD operations)

```

**Key insight:** Dependencies flow **inward**. `handlers` depends on `db`, but `db` never imports `handlers`. This is the **Dependency Rule** from Clean Architecture — and you're following it naturally.

## 4. Request Lifecycle — End to End

Here's what happens when a user creates an entry (`POST /entries`):

```
Client sends: POST /entries
  Headers: Authorization: Bearer <jwt-token>
  Body: {"text": "Great day", "mood": 8, "category": "personal"}

Step 1: RequestIDMiddleware
  → Generates random 16-char hex ID (e.g., "a1b2c3d4e5f67890")
  → Stores in context: ctx = context.WithValue(ctx, "request_id", id)
  → All downstream logs include this ID

Step 2: TimeoutMiddleware
  → Creates context.WithTimeout(ctx, 10s)
  → Runs handler in goroutine
  → select: handler finishes OR 10s timeout → 504

Step 3: MetricsMiddleware
  → Records request start time
  → Wraps ResponseWriter to capture status code
  → After handler: records duration, status, endpoint

Step 4: RateLimitMiddleware
  → Extracts IP from r.RemoteAddr
  → Redis INCR ratelimit:<ip>
  → count > 100? → 429 Too Many Requests
  → count == 1? → Set TTL (60s window)

Step 5: LoggingMiddleware
  → Records start time
  → Calls handler
  → Logs: method, path, duration_ms with request_id

Step 6: AuthMiddleware
  → Extracts "Bearer <token>" from Authorization header
  → Validates JWT signature + expiration
  → Extracts user_id from claims
  → Stores in context: ctx = context.WithValue(ctx, "user_id", int64(userID))

Step 7: CreateEntry handler
  → Reads user_id from context
  → Decodes JSON body → CreateEntryRequest struct
  → Validates: text not empty, mood 1-10, category not empty
  → db.InsertEntry() → SQL INSERT with parameterized query
  → cache.Delete("count:user:<id>") → Invalidate pagination count cache
  → worker.AddJob("entry_created", payload) → Async background processing
  → respondJSON(w, 201, response)

Step 8: Response flows back through middleware (logging records duration)

Client receives: 201 Created
  {"success": true, "message": "Entry created successfully", "id": 42}
```

## 5. Middleware Chain Architecture

### The Pattern: Higher-Order Functions

Every middleware follows the **same signature**:

```go
func SomeMiddleware(next http.HandlerFunc) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        // Before: do pre-processing
        next(w, r)  // Call the next handler
        // After: do post-processing
    }
}
```

This is a **higher-order function** — a function that takes a function and returns a function. It's the Go equivalent of Python decorators or JavaScript middleware.

### Chain Order and Why It Matters

```
RequestID → Timeout → Metrics → RateLimit → Logging → [Auth] → Handler
(outermost)                                                    (innermost)
```

| Position | Middleware | Reason |
|----------|-----------|--------|
| 1st | RequestID | Must be first — every log line needs this ID for tracing |
| 2nd | Timeout | Kills stuck requests early — protects server resources |
| 3rd | Metrics | Tracks ALL requests including timeouts and rate-limited ones |
| 4th | RateLimit | Reject abusers before doing expensive work (DB queries, auth) |
| 5th | Logging | Logs request details after rate limit check |
| 6th | Auth | Only on protected routes — validates JWT, adds user_id to context |

### Context as a Data Pipeline

The middleware chain uses `context.Context` as a **data pipeline**:

```
RequestIDMiddleware adds:  "request_id" → "a1b2c3d4"
TimeoutMiddleware adds:    deadline (10s from now)
AuthMiddleware adds:       "user_id" → int64(3)

Handler reads:
  userID := r.Context().Value("user_id").(int64)
  requestID := r.Context().Value("request_id").(string)
```

**Go concept learned here:** Context is the idiomatic way to pass request-scoped data across function boundaries without adding parameters to every function signature.

## 6. Error Handling Patterns

### Pattern 1: Immediate Check, Early Return

This is the **most important Go pattern** and you use it everywhere:

```go
result, err := someFunction()
if err != nil {
    // Handle error immediately
    return  // Exit early
}
// Happy path continues
```

**Example from your code** (`auth.go`):
```go
passwordHash, err := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)
if err != nil {
    slog.Error("Error hashing password", "error", err)
    errorResponseAuth(w, http.StatusInternalServerError, "Failed to process password")
    return
}
```

### Pattern 2: Sequential Validation with Early Returns

Your handlers validate inputs one at a time, rejecting early:

```go
if req.Email == "" {
    errorResponseAuth(w, http.StatusBadRequest, "email is required")
    return
}
if !strings.Contains(req.Email, "@") {
    errorResponseAuth(w, http.StatusBadRequest, "invalid email format")
    return
}
if req.Password == "" {
    errorResponseAuth(w, http.StatusBadRequest, "password is required")
    return
}
```

**Why this is good:** Each check has a clear, specific error message. The user knows exactly what's wrong. No nested if-else pyramid.

### Pattern 3: Error Wrapping with %w

```go
return fmt.Errorf("operation failed after %d attempts: %w", maxAttempts, lastErr)
```

The `%w` verb wraps the original error so callers can use `errors.Is()` or `errors.As()` to inspect the chain:

```go
if errors.Is(err, jwt.ErrTokenExpired) {
    // Handle specifically
}
```

### Pattern 4: Fail Open vs. Fail Closed

Your rate limiter **fails open** (if Redis is unavailable, allow the request):
```go
count, err := redis.Client.Incr(context.Background(), redisKey).Result()
if err != nil {
    return true  // Redis down → allow request (fail open)
}
```

**Trade-off:** Availability over strictness. Better to let some extra requests through than to block everyone because Redis went down.

### Pattern 5: Don't Leak Internal Errors

```go
// Login: Don't reveal if user exists
slog.Warn("Login attempt for non-existent user", "email", req.Email)
errorResponseAuth(w, http.StatusUnauthorized, "Invalid email or password")
```

The log has details (for debugging), but the client gets a generic message (for security).

## 7. Concurrency Patterns (What You Built)

### 7.1 Worker Pool (Goroutines + Channels)

**File:** `internal/worker/worker.go`

```
                    ┌─────────────┐
AddJob() ──push──→ │  JobQueue    │ ←── buffered chan Job (cap: 100)
                    │  (channel)   │
                    └──┬──┬──┬────┘
                       │  │  │
                  ┌────┘  │  └────┐
                  ▼       ▼       ▼
              Worker1  Worker2  Worker3   ← goroutines (range over channel)
              (go)     (go)     (go)
```

**Concepts demonstrated:**
- **Buffered channels**: `make(chan Job, 100)` — up to 100 jobs can queue before blocking
- **Fan-out pattern**: One channel, multiple consumers (workers)
- **Non-blocking send**: `select { case JobQueue <- job: ... default: ... }` — drops job if queue full instead of blocking the HTTP handler
- **`range` over channel**: `for job := range JobQueue` — automatically blocks waiting for next job, exits when channel is closed

**Why this matters:** The HTTP handler (`CreateEntry`) returns in milliseconds. The slow work (notifications, analytics) happens in the background. User doesn't wait.

### 7.2 Graceful Shutdown (Channels + Signals)

**File:** `cmd/server/main.go`

```go
quit := make(chan os.Signal, 1)
signal.Notify(quit, os.Interrupt)
<-quit  // Block until Ctrl+C
```

**Concepts demonstrated:**
- **OS signal handling**: Convert OS signals into channel events
- **Blocking receive**: `<-quit` parks the main goroutine until a signal arrives
- **`context.WithTimeout`**: Gives in-flight requests 5 seconds to finish
- **Ordered cleanup**: Stop accepting requests → wait for running ones → close cache → close Redis → close DB

### 7.3 Timeout Middleware (Goroutines + Select)

**File:** `internal/handlers/timeout.go`

```go
done := make(chan struct{})
go func() {
    next(w, r)
    close(done)
}()
select {
case <-done:       // Handler finished
case <-ctx.Done(): // Timeout expired → 504
}
```

**Concepts demonstrated:**
- **`select` statement**: Waits for whichever event happens first
- **Signal channel**: `chan struct{}` used purely for signaling (no data)
- **`close(done)`**: Unblocks all receivers on the channel
- **Context cancellation propagation**: If timeout fires, `ctx.Done()` also triggers in downstream DB/Redis calls that use `ctx`

### 7.4 In-Memory Cache Cleanup (Background Goroutine + Ticker)

**File:** `internal/cache/cache_in_memory.go`

```go
go c.cleanupLoop()  // Started in NewCache()

func (c *Cache) cleanupLoop() {
    ticker := time.NewTicker(30 * time.Second)
    defer ticker.Stop()
    for {
        select {
        case <-ticker.C:      // Every 30s → delete expired entries
        case <-c.stopCleanup: // Shutdown signal → exit goroutine
        }
    }
}
```

**Concepts demonstrated:**
- **`time.NewTicker`**: More accurate than `time.Sleep` in a loop (compensates for processing time)
- **`sync.RWMutex`**: Read-lock for `Get()` (multiple readers), write-lock for `Set()`/`Delete()`/cleanup (exclusive)
- **Stop channel**: `close(c.stopCleanup)` gracefully terminates the goroutine

### 7.5 Circuit Breaker (Mutex-Protected State Machine)

**File:** `internal/circuitbreaker/circuitbreaker.go`

```
 CLOSED ──(5 failures)──→ OPEN ──(30s cooldown)──→ HALF-OPEN
   ▲                        ▲                          │
   │                        │                          │
   └──(success)─────────────┴──────(failure)───────────┘
            ◄────(test success)────┘
```

**Concepts demonstrated:**
- **`sync.Mutex`**: Protects state transitions from concurrent goroutines
- **State machine pattern**: Three states with well-defined transitions
- **Higher-order function**: `Execute(func() error)` — the breaker doesn't know what it's protecting; it's generic

## 8. Concurrency Opportunities (What You Could Add)

### 8.1 Parallel Health Checks

**Current:** Sequential (DB ping, then Redis ping)
**Better:** Check both simultaneously

```go
func HealthHandler(w http.ResponseWriter, r *http.Request) {
    var dbErr, redisErr error
    var wg sync.WaitGroup

    wg.Add(2)
    go func() {
        defer wg.Done()
        dbErr = db.DB.Ping()
    }()
    go func() {
        defer wg.Done()
        redisErr = redis.Client.Ping(context.Background()).Err()
    }()
    wg.Wait()

    // Build response from dbErr, redisErr
}
```

**Benefit:** If DB ping takes 50ms and Redis takes 30ms — sequential = 80ms, parallel = 50ms.

### 8.2 Context-Aware Database Queries

**Current:** `DB.Exec(query, args...)` — no timeout awareness
**Better:** `DB.ExecContext(ctx, query, args...)` — respects request timeout

```go
func InsertEntry(ctx context.Context, userID int, text string, mood int, category string) (int64, error) {
    query := `INSERT INTO entries (user_id, text, mood, category) VALUES (?, ?, ?, ?)`
    result, err := DB.ExecContext(ctx, query, userID, text, mood, category)
    // ...
}
```

**Benefit:** If the timeout middleware fires after 10s, the DB query is cancelled too — no orphaned queries.

### 8.3 Concurrent Cache Invalidation + DB Write

**Current:** Sequential: insert → cache delete → worker enqueue
**Possible:** Cache delete and worker enqueue are independent of each other

```go
id, err := db.InsertEntry(...)
if err != nil { return }

// These two operations don't depend on each other
go cache.Delete(fmt.Sprintf("count:user:%d", userID))
worker.AddJob("entry_created", payload)
```

**Note:** `worker.AddJob` is already non-blocking (uses `select`), so this is minor.

### 8.4 Batch Metrics Flushing

**Current:** Each request individually updates the Metrics struct under a mutex lock
**Future:** Buffer metrics and flush periodically (reduces lock contention under high load)

This is what Prometheus client libraries do internally.

### 8.5 Connection Pooling Tuning

SQLite's `database/sql` already pools connections, but you could tune it:

```go
DB.SetMaxOpenConns(25)
DB.SetMaxIdleConns(5)
DB.SetConnMaxLifetime(5 * time.Minute)
```

## 9. Idiomatic Go Patterns

### 9.1 Patterns You're Using Correctly

| Pattern | Where | Example |
|---------|-------|---------|
| **`err != nil` early return** | Everywhere | Every function checks errors immediately |
| **`defer` for cleanup** | `db.go`, `main.go` | `defer rows.Close()`, `defer db.CloseDB()` |
| **Blank import `_`** | `db.go` | `_ "modernc.org/sqlite"` — driver registration |
| **Exported = Uppercase** | All packages | `Config`, `InsertEntry`, `RateLimitRequests` |
| **Unexported = lowercase** | Internal helpers | `createTables`, `errorResponse`, `processJob` |
| **`context.Context` passing** | Middleware chain | Request ID, user ID, timeout via context |
| **`json` struct tags** | `models.go`, request structs | `json:"field"`, `json:"-"`, `json:"field,omitempty"` |
| **`http.HandlerFunc` adapter** | All middleware | Higher-order functions returning `http.HandlerFunc` |
| **Pointer receivers** | `Cache`, `Metrics` | `func (c *Cache) Get(...)` — modifies state |
| **Table-driven `switch`** | `worker.go` | `switch job.Type { case "entry_created": ... }` |
| **Buffered channels for async** | `worker.go` | `make(chan Job, 100)` |
| **`select` for multiplexing** | `timeout.go`, `cache_in_memory.go` | Wait for first of N channel events |

### 9.2 Variable Naming Conventions

Your code follows Go's naming convetions well:
- **Short names in small scope**: `i`, `p`, `l` in loop variables
- **Descriptive in larger scope**: `userID`, `passwordHash`, `rateLimitWindow`
- **`err` for errors**, **`ok` for comma-ok form** — used consistently
- **`req` for requests**, **`cfg` for config** — standard abbreviations
- **`w` for ResponseWriter**, **`r` for Request** — universal Go HTTP convention

### 9.3 The `interface{}` → `any` Evolution

Your code uses `interface{}` (e.g., `map[string]interface{}`). Since Go 1.18, `any` is an alias for `interface{}`. Both work, but modern Go prefers `any`:

```go
// Current (works fine)
map[string]interface{}

// Modern Go (same thing, just shorter)
map[string]any
```

## 10. Anti-Patterns & Improvement Opportunities

These aren't bugs — they're areas where the code could become more robust or idiomatic as you grow.

### 10.1 String Context Keys

**Current:**
```go
ctx := context.WithValue(r.Context(), "user_id", int64(userID))
```

**Problem:** If two packages both use the string `"user_id"`, they'd collide. Go's convention is to use unexported custom types:

```go
type contextKey string
const userIDKey contextKey = "user_id"

ctx := context.WithValue(r.Context(), userIDKey, int64(userID))
userID := r.Context().Value(userIDKey).(int64)
```

### 10.2 Global Variables (`var DB *sql.DB`, `var Client *redis.Client`)

**Current:** Packages expose global mutable state.

**Why it works now:** Single server, single connection. Simple and readable.

**Future concern:** Makes testing harder (can't swap in a mock DB). The production evolution is **dependency injection**:

```go
type EntryHandler struct {
    db    *sql.DB
    cache CacheInterface
}

func (h *EntryHandler) CreateEntry(w http.ResponseWriter, r *http.Request) {
    // Use h.db instead of global db.DB
}
```

### 10.3 `map[string]interface{}` Instead of Typed Structs for DB Results

**Current:**
```go
entry := map[string]interface{}{
    "id": id, "user_id": userIDResult, "text": text, ...
}
```

**Better:** Use the `models.Entry` struct you already defined:
```go
entry := models.Entry{
    ID: id, UserID: userIDResult, Text: text, ...
}
```

**Why:** Compile-time type safety. Typo in `"usr_id"` vs `"user_id"` is a runtime bug with maps, but a compile error with structs.

### 10.4 Circuit Breaker Holds Lock During Operation Execution

**Current:** `cb.mu.Lock()` is held while `operation()` runs (which could be a slow Redis call).

**Impact:** All other goroutines calling `Execute()` are blocked until the operation completes — serializing Redis calls.

**Production fix:** Lock only for state checks, unlock before calling the operation, re-lock to update state. More complex, but avoids bottleneck.

### 10.5 No Unit Tests

Testing is done via PowerShell scripts (`test-endpoints/`). This is fine for integration testing, but Go's built-in test framework is a core skill:

```go
// db_test.go
func TestInsertEntry(t *testing.T) {
    // Setup in-memory DB
    // Insert entry
    // Assert result
}
```

## 11. Security Architecture

### Authentication Flow

```
Register: password → bcrypt hash (cost=10) → stored in DB
Login:    password → bcrypt compare with hash → JWT token (24h expiry)
Request:  JWT in header → validate signature → extract user_id → context
```

### Security Decisions (Good Ones)

| Decision | Implementation | Why It Matters |
|----------|----------------|----------------|
| Password never stored in plain text | `bcrypt.GenerateFromPassword` | Even if DB is stolen, passwords are safe |
| `json:"-"` on PasswordHash | `models.User` struct | Password hash never accidentally serialized to JSON |
| Parameterized SQL queries | `?` placeholders everywhere | Prevents SQL injection |
| Generic error messages on login | "Invalid email or password" | Prevents user enumeration |
| JWT secret from environment | `os.Getenv("JWT_SECRET")` | Never hardcoded in source |
| Auth middleware on protected routes only | `/entries` has auth, `/health` doesn't | Principle of least privilege |
| User can only see own entries | `WHERE user_id = ?` in every query | Data isolation by design |
